REST: REpresentational State Transfer
software architecture style to guide design of architecture for the web.

RE : Representational (format)
S : State (Data)
T : Transfer (transfer data b/w two parties)

server is producer, client is consumer.

1)client & server should be independent, client should not know
that in which language the APIs in the server are written and vice versa.


2) Stateless: do not store any data in server. it should be stateless

3) Cacheable: client can cache and reuse it.

4) Layered architecture: load balancing by scalibility

5) Uniform architecture: all interactions in unified way.

6) Code on demand: client can download it and use server code !




REST Concepts:

* Resource : anything we want to expose to outside world, through our application
Library management system, Book & User
Student Management system, Student, teacher, class , subject


* Sub-Resource :
something that depends on main rsource.
eg. : comments inside post.
laptops inside Student.


* URI : Uniform resource identifier,
used to idenfity resource


* Http Methods: defines set of request methods to indicate desired action
to be performed on a given resource. like GET,PUT,DELETE.


* Http Response codes: whether a HTTP Request has been successfully completed.

when user is already logged in=>
The SecurityContext is checked/restored in the filter chain,
before the request reaches AuthenticationManager.

AuthenticationManager is only involved when
authentication is actually needed (such as a login attempt).



  Login
    ↓
  AuthenticationManager
    ↓
  SecurityContext created
    ↓
  SecurityContext saved in HTTP Session
    ↓
  Future requests
    ↓
  Session restored
    ↓
  SecurityContextHolder populated
    ↓
  User recognized instantly









  For a normal request (GET /profile, GET /posts, etc.)

  Every request goes through the filter chain, and Spring checks whether a SecurityContext already exists for that user.

  GET /profile
      ↓
  Filter Chain
      ↓
  Load SecurityContext from Session
      ↓
  Authentication found?
      ↓
  Yes → User is already logged in

  No call to AuthenticationManager.
  No password verification.
  No loadUserByUsername().

  For a login request (POST /login)

  The filter chain sees that this is the login endpoint and triggers authentication.

  POST /login
      ↓
  UsernamePasswordAuthenticationFilter
      ↓
  AuthenticationManager
      ↓
  DaoAuthenticationProvider
      ↓
  UserDetailsService

  Here Spring is verifying credentials.

  So what happens if John is already logged in and sends another normal request?
  GET /posts

  Spring does:

  Filter Chain
      ↓
  Restore SecurityContext
      ↓
  Found John
      ↓
  Allow request

  No database lookup.

  What happens if John explicitly submits the login form again?
  POST /login
  email=john@gmail.com
  password=1234

  Spring treats it as a fresh login attempt:

  UsernamePasswordAuthenticationFilter
      ↓
  AuthenticationManager
      ↓
  Password Check

  Even if John was already logged in.

  The key distinction

  Every request:

  Check/restore SecurityContext

  Only login requests (or other authentication flows):

  Call AuthenticationManager

  So the check of the existing authenticated user happens on every secured request,
  while the expensive authentication process (database lookup + password verification)
  only happens when authentication is actually being performed.
